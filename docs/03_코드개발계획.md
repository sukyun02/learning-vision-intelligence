### B-1. PyramidNet-272 α=200 구현 — [models/pyramidnet.py](../models/pyramidnet.py)

|---|---|
| Depth | 272 |

| Total blocks | 90 (= (272-2)/3) |
| Initial channels | 16 |
| Final channels | 216 |
| Parameters | ~28.2M |
| ShakeDrop | linear schedule p: 0 → 0.5 |

### B-2. ShakeDrop

- Forward: `(gate + α - gate·α) × x`, α ∈ [-1, 1]
- Backward: `(gate + β - gate·β) × grad`, β ∈ [0, 1]
- Drop 확률은 블록 인덱스에 비례 (얕은 → 0, 깊은 → 0.5)
- `model.eval()` 시 비활성화

## Phase C. Loss / Augmentation / Metrics (✅ 완료)

### C-1. HierarchicalLoss — [losses/hierarchical_loss.py](../losses/hierarchical_loss.py)

```
L = (1 - λ) · CE(fine_logits, fine) + λ · NLL(coarse_logits, coarse)
λ = 0.8
```

- 100→20 매핑 행렬을 곱해 coarse log-prob 계산
- CutMix-aware: `lam < 1.0`이면 두 라벨 loss를 가중 평균

### C-2. Augmentation — [data/cifar100.py](../data/cifar100.py)

- RandomCrop(32, padding=4), RandomHorizontalFlip
- **AutoAugment (CIFAR-10 policy)**
- **Cutout (16px)**
- **CutMix (α=1.0, p=0.5)** — DataLoader collate 단계에서 적용
- ShakeDrop은 모델 내부

### C-3. Super-Class 매핑

- `FINE_TO_COARSE` 리스트 + `fine_to_coarse_tensor()` 헬퍼
- CIFAR-100 메타데이터 기반 100→20 매핑

### C-4. Metrics — [evaluate.py](../evaluate.py)

- Fine Top-1 Accuracy
- Super-Class Accuracy
- TTA: 원본 + horizontal flip 2-view 평균 ([evaluate.py:97-124](../evaluate.py))
- **Super-class re-scoring**: P(fine) × P(super|fine) ([evaluate.py:58-73](../evaluate.py))

> ⚠️ **dead code 정리 필요**: [evaluate.py:33-51, 87-95](../evaluate.py)에 4-corner crop TTA 함수가 정의되어 있으나 실제 평가 루프에서는 호출되지 않는다. 원본 계획서의 "hflip + 4-corner" 구현 의도가 hflip만 남은 형태. 향후 정리 권고.

## Phase D. Config & 하이퍼파라미터 (⚠️ VRAM 조정 필요)

### D-1. 현재 [.env](../.env) — 33.78% 실험과의 비교 포함

| 변수 | 33.78% 실험 (slide p3) | 코드 (이전) | **권고 (확정)** | 사유 |
|---|---|---|---|---|
| EPOCHS | 200 | 1000 | 1000 유지 | 미수렴 주원인은 epoch 부족이 아님 |
| **BATCH_SIZE** | 64 | **256** | **64** | bs=256→~20GB+ / bs=64→~6-8GB |
| **LR** | **0.2** (표준의 4×) | 0.1 (표준의 0.5×) | **0.05** | PyramidNet/ShakeDrop 원논문 reference (bs=128, lr=0.1)에서 linear scaling |
| **WEIGHT_DECAY** | 5e-4 | 5e-4 | **1e-4** | ShakeDrop 원논문 값. ShakeDrop+Cutout 정규화 중첩 완화 |
| SWA_EPOCHS | (없음) | 250 | 250 유지 | epoch 75% 이후 SWA |
| NUM_WORKERS | — | 6 | 6 | |
| AMP | (없음) | ON | ON | 33.78% 실험 후 추가됨 |
| AutoAugment+Cutout | (없음, CutMix만) | ON | ON | 33.78% 실험 후 추가됨 |

### D-1-1. 33.78% 미수렴 원인 진단

원본 실험은 PyramidNet-272를 lr=0.2 / bs=64로 돌렸다 (slide p3). lr/sample = 3.1e-3.

PyramidNet·ShakeDrop 원논문 표준은 bs=128 / lr=0.1, 즉 lr/sample = **7.8e-4**.

→ 실험 config는 **표준의 4배 LR**이었고, ShakeDrop의 강한 stochastic perturbation과 결합되어 loss landscape에서 진동했을 가능성이 매우 높다. PyramidNet-272는 200ep만 돌려도 보통 70% 이상 나오므로, 33.78%는 "epoch 부족"이 아니라 **학습 자체가 깨졌다는 신호**다. epoch을 늘려도 같은 LR로는 못 고친다.

> ⚠️ 분석서(`PyramidNet_현황분석_개선계획서.md`)는 "lr=0.2를 0.4로 더 올리거나 sweep"을 권고했지만, 이는 깨진 baseline 위에 또 4× scaling을 적용하는 것이고 표준에서 8× 멀어진다. **따르지 말 것.**

### D-2. VRAM 절감 전략 분석

PyramidNet-272 + ShakeDrop + BN 조합 특성을 따져본 결과, **bs 축소가 가장 안전**하다.

| 옵션 | PyramidNet+ShakeDrop과의 궁합 | 판정 |
|---|---|---|
| **A. bs 256→64** | PyramidNet 원논문 single-GPU 기본값. ShakeDrop 자체가 강한 stochastic noise라서 BN noise 영향이 작음. LR linear scaling 정확히 적용하면 정확도 손실 없음. | ✅ **1차 권고** |
| B. α 200→150/120 | 파라미터 ~30% 감소. 정확도 1-2%p 하락 가능 → 84% 마진이 빠듯. | △ A 실패 시 fallback |
| C. Gradient accumulation | **BN 통계는 micro-batch 단위로 계산**됨 → effective bs와 동등하지 않음. ShakeDrop+BN 조합에서는 의미가 약함. | ❌ 비추천 |
| D. Gradient checkpointing | 메모리 35-40% 절감. 단, forward 재실행 시 ShakeDrop의 random α가 backward와 불일치할 위험. RNG state save/restore 필요. | △ 마지막 수단 |

**확정 절차**:
1. [.env:19](../.env) `BATCH_SIZE=256 → 64`
2. [.env:20](../.env) `LR=0.1 → 0.025`
3. `python train.py --epochs 1` 1-epoch 스모크로 VRAM 확인
4. 8-12GB 안에 들어오면 본 학습 진행
5. 안 들어오면 → α=170으로 후퇴 ([models/pyramidnet.py:179-181](../models/pyramidnet.py)에 alpha 인자 추가 필요)

### D-3. (선택) Warmup 짧게

작은 배치는 warmup 영향이 작음. [train.py:221](../train.py)의 `warmup_epochs=5`를 3으로 줄여도 됨.

## Phase E. 디버그 사전검증 (❌ 미구현 — 권고 추가)

> 원본 계획서의 Karpathy 4-stage는 **최종 성능에 영향을 주지 않지만**, 현재 baseline이 33.78%로 미수렴 상태이므로 **LR sweep 1회는 필수적**이다. 본 학습(1000ep × 며칠) 전에 1시간 들여 LR을 검증하면 큰 비용을 줄일 수 있다.

### E-1. 권고 구현

[train.py](../train.py)에 `--mode` CLI 플래그 1개만 추가:

```python
parser.add_argument("--mode", choices=["train", "overfit_batch", "lr_sweep"],
                    default="train")
```

#### Stage 1: `--mode overfit_batch`
- 1 batch 고정, 500 step, augmentation OFF, ShakeDrop OFF
- 기대: loss → 0, accuracy → 100%
- 실패 시: forward/backward 연결, label shape, LR 점검
- ~2분 소요

#### Stage 2: `--mode lr_sweep`
- LR ∈ {1e-4, 3e-4, 1e-3, 3e-3, 1e-2, 3e-2, 1e-1, 3e-1, 1.0} (9개)
- 각 LR로 10ep만 학습
- WandB에서 LR vs val_acc 비교
- ~1시간 소요
- 결과를 [.env](../.env)의 LR에 반영

작업량: ~50줄 수정. 별도 스크립트 X.

### E-2. (선택사항) Stage 3 = baseline (no aug)

- augmentation/ShakeDrop 모두 끄고 100ep → augmentation 효과 정량화
- 본인 모델만 가지고 디버깅할 때 유용
- **필수는 아님**
